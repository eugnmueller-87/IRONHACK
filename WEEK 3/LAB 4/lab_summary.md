# Lab Summary — Using MCP in LangChain

MCP (Model Context Protocol) and direct API integration both let a LangChain agent access external data, but they differ in scope and maintenance cost: with direct API integration you write custom Python code for every data source (open the file, call the REST endpoint, parse the response), which is fast and simple for a single integration but does not scale — each new source requires new glue code with its own error handling and interface conventions. MCP inverts this: the data source ships as a self-describing server that exposes standardised tools and resources, so the agent discovers and calls them dynamically without any integration-specific code in the application layer; the trade-off is the overhead of running an MCP server process (in this lab, an `npx` child process) and the extra abstraction layer, which adds startup latency and a small debugging surface. In practice, prefer direct API integration when you control a single, stable data source and need raw performance or fine-grained API access; choose MCP when you need to connect to multiple systems, want your integrations to be reusable across different agent frameworks (LangChain, CrewAI, Claude Desktop, etc.), or expect the set of tools to grow over time — the upfront cost of the MCP server pays off quickly as soon as a second consumer or a second data source appears.
