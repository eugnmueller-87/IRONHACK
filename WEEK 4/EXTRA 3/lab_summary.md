# Lab Summary — Error Handling and Scheduled Workflows

For the error-handling workflow (`daily_data_fetcher.json`), the HTTP Request node uses n8n's built-in `continueErrorOutput` mode so that failures route to a dedicated error branch instead of halting silently; the node is also configured with 3 retries and a 5-second delay to absorb transient faults like network timeouts or 5xx responses, while permanent errors (4xx) simply pass through the error branch without wasting retries. For the scheduled workflow (`daily_summary_generator.json`), idempotency is guaranteed by stamping every record with the current UTC date (`$now.toISO().split('T')[0]`) at the start of the run and then passing that value through the entire chain; an IF node compares the record's `record_id` against today's date and routes to a no-op "Skip" branch if a record already exists, so running the workflow twice on the same day never produces duplicate entries. The schedule is set to 9 AM daily via the cron expression `0 9 * * *`, chosen because it gives overnight data a full processing window before the business day starts and is easy to verify in execution history.
